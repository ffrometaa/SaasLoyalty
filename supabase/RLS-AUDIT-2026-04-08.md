# RLS Security Audit — LoyaltyOS
**Fecha:** 2026-04-08
**Tablas analizadas:** 39
**Migration de corrección:** `20260408000010_fix_gamification_engine_rls.sql`

---

## Resumen ejecutivo

El audit identificó dos categorías de problemas reales:

1. **`stripe_events` sin RLS** — tabla expuesta a lectura/escritura pública vía `anon key`.
2. **9 tablas del gamification engine + `member_invitations` con policies rotas** — usan `auth_tenant_id()` que siempre retorna `NULL`, haciendo que las policies sean no-ops. Dos de esas tablas además exponían datos cross-tenant a cualquier usuario autenticado.

El resto del schema (28 tablas) está correctamente protegido con `current_tenant_id()` / `current_member_id()` desde la migration `20260406000006_fix_rls_all_tables.sql`.

---

## Causa raíz

La migration inicial (`001_initial_schema.sql`) definió dos helpers:

```sql
-- ROTO — lee session variable que nunca se setea desde middleware
CREATE OR REPLACE FUNCTION auth_tenant_id()
RETURNS UUID AS $$
  SELECT NULLIF(current_setting('app.tenant_id', true), '')::UUID;
$$ LANGUAGE SQL STABLE;

CREATE OR REPLACE FUNCTION auth_member_id()
RETURNS UUID AS $$
  SELECT NULLIF(current_setting('app.member_id', true), '')::UUID;
$$ LANGUAGE SQL STABLE;
```

Estas funciones dependen de `current_setting('app.tenant_id')` que **nunca se setea desde ningún middleware**. Siempre retornan `NULL`.

La migration `20260406000006_fix_rls_all_tables.sql` introdujo las versiones correctas:

```sql
-- CORRECTO — SECURITY DEFINER lookup real via auth.uid()
CREATE OR REPLACE FUNCTION current_tenant_id()
RETURNS UUID LANGUAGE sql SECURITY DEFINER STABLE AS $$
  SELECT COALESCE(
    (SELECT id        FROM tenants      WHERE auth_user_id = auth.uid() AND deleted_at IS NULL LIMIT 1),
    (SELECT tenant_id FROM tenant_users WHERE auth_user_id = auth.uid() LIMIT 1),
    (SELECT tenant_id FROM members      WHERE auth_user_id = auth.uid() LIMIT 1)
  );
$$;

CREATE OR REPLACE FUNCTION current_member_id()
RETURNS UUID LANGUAGE sql SECURITY DEFINER STABLE AS $$
  SELECT id FROM members WHERE auth_user_id = auth.uid() LIMIT 1;
$$;
```

Esa migration corrigió las tablas básicas pero **no incluyó las tablas del gamification engine ni `member_invitations`**.

## Problemas encontrados

### CRÍTICO-1 — `stripe_events`: sin RLS

| Campo | Valor |
|---|---|
| Tabla | `stripe_events` |
| Estado | `rowsecurity: false` |
| Riesgo | Cualquier usuario con `anon key` puede hacer SELECT/INSERT/UPDATE/DELETE |
| Datos expuestos | Stripe event IDs, tipos de evento, timestamps de pagos |
| Fix | `ALTER TABLE stripe_events ENABLE ROW LEVEL SECURITY;` (sin policies = bloqueado para anon/authenticated; service_role bypasea RLS) |

El webhook (`apps/web/app/api/webhooks/stripe/route.ts`) usa `createServiceRoleClient()` — no se ve afectado.

---

### CRÍTICO-2 — `missions`: policy member sin filtro de tenant

| Campo | Valor |
|---|---|
| Tabla | `missions` |
| Policy rota | `member_missions_read` |
| Definición actual | `FOR SELECT USING (is_active = true)` |
| Riesgo | Cualquier usuario autenticado ve misiones activas de TODOS los tenants |
| Policy de admin también rota | `tenant_missions_rls` usa `auth_tenant_id()` → NULL → no-op → admins tampoco pueden ver sus propias misiones |

---

### CRÍTICO-3 — `point_multipliers`: policy member sin filtro de tenant

| Campo | Valor |
|---|---|
| Tabla | `point_multipliers` |
| Policy rota | `member_multipliers_read` |
| Definición actual | `FOR SELECT USING (is_active = true AND now() BETWEEN starts_at AND ends_at)` |
| Riesgo | Cualquier usuario autenticado ve multiplicadores activos de todos los tenants |
| Policy de admin también rota | `tenant_multipliers_rls` usa `auth_tenant_id()` → NULL → no-op |

---

### CRÍTICO-4 — Gamification engine: policies no-ops en 9 tablas

Todas usan `auth_tenant_id()` o `auth_member_id()` → siempre `NULL` → ninguna row pasa el filtro.

| Tabla | Policies rotas | Efecto real |
|---|---|---|
| `member_behavior_scores` | `tenant_behavior_scores_rls`, `member_scores_self_read` | Nadie puede leer nada |
| `missions` | `tenant_missions_rls` | Admins no ven sus misiones; solo `member_missions_read` actúa (sin tenant filter) |
| `mission_steps` | `tenant_mission_steps_rls` | Nadie puede leer nada |
| `member_mission_progress` | `tenant_mission_progress_rls`, `member_mission_progress_self_read` | Nadie puede leer nada |
| `point_multipliers` | `tenant_multipliers_rls` | Admins no ven; solo `member_multipliers_read` actúa (sin tenant filter) |
| `leaderboard_snapshots` | `tenant_leaderboard_rls`, `member_leaderboard_read` | Nadie puede leer nada |
| `engine_activity_log` | `tenant_engine_log_rls` | Nadie puede leer nada |
| `churn_interventions` | `tenant_interventions_rls` | Nadie puede leer nada |
| `dynamic_challenges` | `tenant_dynamic_challenges_rls`, `member_dynamic_challenges_self_read` | Nadie puede leer nada |

---

### MENOR — `member_invitations`: policy no-op

| Campo | Valor |
|---|---|
| Tabla | `member_invitations` |
| Policy rota | `tenant_manage_invitations` — usa `auth_tenant_id()` → NULL |
| Impacto funcional | Ninguno — ambos API routes usan `createServiceRoleClient()` que bypasea RLS |
| Fix | Consistencia — unificar bajo `current_tenant_id()` |

## Lo que está bien

- **28 tablas** correctamente protegidas con `current_tenant_id()` / `current_member_id()`.
- `tenant_plan_history`: bloqueada totalmente (`USING (false)`) — solo `service_role`.
- `demo_requests`: INSERT anónimo permitido correctamente; lectura solo vía `service_role`.
- `impersonation_logs` y `super_admins`: protegidas con `is_super_admin()`.
- `tenant_users`: protección anti-auto-borrado del owner.
- `stripe_events` webhook: usa `service_role` → habilitar RLS no lo rompe.
- `legal_documents` con `USING (true)`: aceptable para términos públicos.
- `current_tenant_id()` y `current_member_id()`: SECURITY DEFINER, resuelven via `auth.uid()` (JWT, no manipulable desde frontend).
